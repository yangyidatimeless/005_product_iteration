### 发言 #3 - 允灿
**时间**: 2026-03-18 23:52 UTC

**我的见解**：

感谢少锋的 QA 视角建议！非常专业，我来逐一回应和补充：

**先回答少锋的问题**：

1. **MVP 自动化测试覆盖率**：目前约 65%。核心算法模块（关键词匹配、评分模型）有单元测试，但 FFmpeg 集成和文件 IO 部分还没覆盖。后续我会补充 pytest 测试用例，目标提升到 85%+。

2. **多主播并发的服务端瓶颈**：
   - **CPU 瓶颈**：Whisper 转录是计算密集型，单路直播约占用 30-40% CPU（使用 small 模型），理论上 4 路并发会吃满 CPU
   - **内存瓶颈**：Whisper 模型加载后约 500MB，每路并发额外占用 200-300MB 音频处理缓冲，8 路并发约需 3-4GB 内存
   - **IO 瓶颈**：视频文件读写和网络下载可能成为瓶颈，建议使用 SSD + 异步 IO
   - **解决方案**：短期用任务队列（Celery）+ 多进程，长期考虑 GPU 加速或分布式部署

**关于产品定位**：
我赞同少锋的**工具型产品优先**策略，从技术角度补充：
- 工具型产品 API 边界清晰，便于服务端模块化设计
- 可以先做 Single-Tenant（单租户）架构，快速验证
- 后续如果要 SaaS 化，再考虑 Multi-Tenant（多租户）改造

**关于功能优先级**：
从技术实现难度和价值的矩阵评估：

| 功能 | 实现难度 | 用户价值 | 优先级 |
|------|----------|----------|--------|
| 自动上传 B 站 | 中（需对接 B 站 API） | 高（闭环核心） | P0 |
| 智能标题/封面 | 中高（需 AI 模型） | 高（提升点击率） | P0 |
| 多平台分发 | 高（各平台 API 差异大） | 中 | P1 |
| 数据分析 | 中（需数据积累） | 中 | P2 |

**建议**：P0 功能做完就可以发布 v1.0，收集用户反馈后再决定 P1/P2 的投入。

**关于商业化模式**：
从技术架构支持角度：
- **SaaS 订阅制** → 需要用户系统、权限管理、计费系统（技术成本最高）
- **按切片数量收费** → 需要用量统计和配额控制（中等复杂度）
- **免费 + 增值** → 需要功能开关和版本管理（较低复杂度）

**我的建议**：先做**免费 + 增值**模式，技术成本最低，可以快速上线验证。基础功能免费（每天 3 个切片），高级功能收费（无限切片 + 智能标题封面）。

**关于技术架构演进**：
我的观点：**先单体，后微服务**。

当前阶段（v1.0）：
```
┌─────────────────────────────────────┐
│  FastAPI 单体应用                   │
│  ├─ 录制模块（直播流下载）          │
│  ├─ 转录模块（Whisper）             │
│  ├─ 切片模块（关键词 + 评分+FFmpeg） │
│  └─ 上传模块（B 站 API）            │
└─────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────┐
│  SQLite 数据库（本地文件）          │
└─────────────────────────────────────┘
```

未来如果需要支持 10+ 主播并发，再考虑拆分：
- 录制服务（独立部署，IO 密集型）
- 转录服务（GPU 节点，计算密集型）
- 切片服务（CPU 密集型）
- 上传服务（网络密集型）
